# Voting-Gated Operations

Privicore can require a **real-time vote** from one or more registered
devices before a sensitive operation is allowed to complete. An
exchange-data-for-token call, for example, can be held at the engine
until a quorum of devices says yes. This is the feature that puts
Privicore in a different category from a plain tokenisation vault: the
authorisation decision is not embedded in a static policy engine, it
happens live, per-operation.

This guide covers the end-to-end setup: registering an authenticator
device, defining a voting configuration, attaching a policy, and
triggering a gated operation. It assumes you've done
[Getting started](/guides/getting-started.html) and understand the
[async command model](/guides/protocol-overview.html#the-async-command-model).

![Voting-gated flow: a data request hits the engine; if a policy is active, a pool issues ballots to registered voter devices; the pool resolves only if the voting configuration's strategy is satisfied, and only then is the operation allowed.](/images/voting-gated-flow.png)

## The pieces

- **Authenticator device.** A registered device that receives ballots
  and replies yes/no. Privicore ships a mobile authenticator ([Android](https://play.google.com/store/apps/details?id=com.privicore.authenticator)
  / [iOS](https://apps.apple.com/us/app/privicore-authenticator/id6736399086))
  that you can use out-of-the-box.
- **Voting configuration.** A named bundle of `{timeLimit, strategy,
  optional deviceIdentifiers}`. `strategy` defines how the pool
  resolves (e.g. `unanimous` — every device must approve). `timeLimit`
  is a per-ballot deadline in seconds.
- **Policy.** A record that links a voting configuration to a set of
  system events (e.g. "when data is exchanged for a token"). When the
  policy is *activated* and the event fires, the engine issues a pool.
- **Pool.** A live voting session spawned by an activated policy
  when the gated event fires. The pool issues ballots over RabbitMQ,
  collects replies, and either resolves (operation proceeds) or
  rejects (operation fails with `command_status: 3`).

## Step 1. Register an authenticator device

Authenticators are regular Privicore devices that have been *promoted*
to voter role. The full chain is four calls:

```bash
# 1. Request a device id under your profile.
curl -X POST {{apiUrl}}/device/request-device-id \
  -H 'X-DPT-AUTHORIZATION: T-MUIFA…' \
  -H 'Content-Type: application/x-www-form-urlencoded' \
  --data-urlencode 'deviceName=phone-1'

# 2. Retrieve the id once the 202 ack lands.
curl {{apiUrl}}/device/retrieve-device-id/{commandId} \
  -H 'X-DPT-AUTHORIZATION: T-MUIFA…'
# → { deviceId: "dev-…" }

# 3. Approve it (owner consent).
curl -X POST {{apiUrl}}/device/approve-device \
  -H 'X-DPT-AUTHORIZATION: T-MUIFA…' \
  --data-urlencode 'deviceId=dev-…'

# 4. Promote the approved device to an authenticator.
curl -X POST {{apiUrl}}/verified-authenticator/register-authenticator \
  -H 'X-DPT-AUTHORIZATION: T-MUIFA…' \
  --data-urlencode 'deviceId=dev-…'
```

At this point the device is eligible to receive ballots. A real
authenticator app picks up the device credentials from
`/device/download-device-configuration` and connects to the engine.

:::info[Mobile authenticator]
If you're using the mobile app, pair it with a QR token generated by
`POST /profile/generate-qr-token-for-authorization` — the app runs the
entire step 1 chain in the background.
:::

## Step 2. Create a voting configuration

The configuration names the strategy and ballot deadline:

```bash
curl -X POST {{apiUrl}}/verified-authenticator/voting-configuration/create \
  -H 'X-DPT-AUTHORIZATION: T-MUIFA…' \
  -H 'Content-Type: application/x-www-form-urlencoded' \
  --data-urlencode 'name=high-assurance-exchange' \
  --data-urlencode 'strategy=unanimous' \
  --data-urlencode 'timeLimit=60'
```

Pass a comma-separated list of device ids in `deviceIdentifiers` to
target specific authenticators.

`strategy: unanimous` means every ballot must come back as approve for
the pool to resolve. Other strategies are defined server-side; query
`/verified-authenticator/retrieve-strategy` for the list available on
your instance.

## Step 3. Attach the voting configuration to a policy

A policy says *which events* are gated by *which voting
configuration*. List the available policy templates to see which
events can be gated on your instance:

```bash
curl {{apiUrl}}/policy/list-policy-templates \
  -H 'X-DPT-AUTHORIZATION: T-MUIFA…'
```

Pick the event id you want to gate, then register and activate the
policy:

```bash
curl -X POST {{apiUrl}}/policy/register-policy \
  -H 'X-DPT-AUTHORIZATION: T-MUIFA…' \
  -H 'Content-Type: application/x-www-form-urlencoded' \
  --data-urlencode 'name=gate-data-exchange' \
  --data-urlencode 'configuration={"scope":"profile"}' \
  --data-urlencode 'votingConfigurationId=<uuid from step 2>' \
  --data-urlencode 'applyingEventIds=<event id from list-policy-templates>'

curl -X POST {{apiUrl}}/policy/activate-policy \
  -H 'X-DPT-AUTHORIZATION: T-MUIFA…' \
  --data-urlencode 'name=gate-data-exchange'
```

`configuration` carries policy parameters specific to the template
you're using — see the template's documentation for what it expects.

## Step 4. Trigger a gated operation

With the policy active, issue the gated operation exactly as you
would any other. The engine detects the active policy at the matching
event, spawns a pool, and holds the operation until the pool
resolves. From your client's perspective:

```bash
# Normal exchange-data-for-token call — no change on the client.
curl -X POST {{apiUrl}}/data-token/exchange-data-for-token \
  -H 'X-DPT-AUTHORIZATION: T-MUIFA…' \
  -H 'Content-Type: application/x-www-form-urlencoded' \
  --data-urlencode 'temporaryTokenSpace=tmp-…' \
  --data-urlencode 'data=<base64 ciphertext>'
# → 202 { commandId }
```

On the WebSocket you'll still receive a single `X-DPT-CAB-ID` message
for your command id — it just arrives later (bounded by the
`timeLimit` you configured). Read `command_status`:

- `2` — pool resolved, operation succeeded as normal.
- `3` — pool rejected (vote failed, timed out, or explicitly denied).
  The message body carries the reason.

Meanwhile, registered authenticators receive ballot notifications and
respond with approve/reject. The Privicore mobile authenticator handles
this out of the box; no client-side work required.

You can watch a pool's progress in real time with
`GET /voting/check-pool-status/{poolId}`, useful for dashboards or
debugging.

## Observing, cancelling, and tearing down

- **Cancel** an in-flight pool (e.g. if business rules change):
  `POST /voting/cancel-pool`.
- **Deactivate** a policy to stop gating future events without losing
  the configuration: `POST /policy/deactivate-policy`.
- **Delete** a voting configuration once no policy references it:
  `POST /verified-authenticator/voting-configuration/delete`.

## What you just did

You registered a device, promoted it to an authenticator, defined a
voting strategy, attached that strategy to a system event via a
policy, and activated the gate. From now on, any call that triggers
the gated event will wait for a real-time vote before completing —
your `exchange-data-for-token` calls, for example, are now decided by
your authenticators in real time rather than by a static allow-list.
